# 영천중앙교회 웹사이트 — 설계 문서

작성일: 2026-06-15
상태: 설계 승인 (구현 플랜 작성 전)

## 1. 배경 / 목적

영천중앙교회(대한예수교장로회, 1956 창립)는 이미 **디모데 앱**이라는 폐쇄형 통합관리 시스템을 운영 중이다(기존 신자만 가입 가능). 본 웹사이트는 그와 별개로, **외부(비신자) 유입 확보**를 위한 **공개 사이트**다.

핵심 비목표(Non-goals):
- 성도 로그인 없음
- 헌금 DB / 성도 관리 없음
- 디모데 앱 기능 재구현 없음

핵심 목표:
- 설교 영상(YouTube) 공개
- 주보를 웹에서 보기 좋게(디자인 렌더) 제공
- 행사 사진 갤러리
- 교회 소식/공지
- 관리자 페이지에서 모든 콘텐츠 관리
- 검색 유입(SEO)을 통한 외부 노출

## 2. 기술 스택

- **프레임워크**: Next.js 16 (App Router) + React 19
- **스타일**: Tailwind CSS v4
- **DB**: Neon Postgres + drizzle ORM (`@neondatabase/serverless`, `drizzle-orm/neon-http`)
- **파일 저장소**: Cloudflare R2 (이미지, hwp 원본, 첨부파일) — `.env`에 `R2_*` 4종 + `R2_PUBLIC_URL`
- **인증**: Neon Auth (Stack Auth 기반)
- **배포**: Vercel (+ Vercel Analytics, Lighthouse CI 이미 설정됨)

### 스택 정리 메모
- 기존 골격에 Supabase 잔재(`supabase/` 디렉토리, admin 코드 내 "Supabase에서 fetch" TODO)가 남아 있음 → **Neon으로 통일**. Supabase 잔재 제거 대상.
- DB 접근은 `src/lib/db/index.ts`의 drizzle 인스턴스로 일원화.

## 3. 인증 모델

- **Neon Auth** 사용. 공개 sign-up **비활성화**.
- 관리자 추가 = Neon Auth 대시보드/서버 API로 수동 생성(email+비번 직접 부여). 외부인 가입 불가.
- `neon_auth.users_sync`는 읽기전용 미러 → 직접 SQL insert 금지.
- **권한 규칙 단순화**: 가입 불가 = 로그인 가능한 사람 = 전원 관리자. 모든 `/admin` 라우트는 "로그인 여부"만 검사(별도 role 분기 없음).
- `/admin/*` 보호는 middleware 레벨에서 세션 검사.

## 5. 주요 기능 설계

### 5.1 설교 (예배·설교)
- 교회 YouTube 채널 영상을 **iframe 임베드**. 영상/오디오 자체 호스팅 안 함(대역폭·스트리밍 부담 회피).
- 예배 종류(`worshipType`)별 필터: **전체 / 주일 / 주일찬양 / 수요** (+ 금요기도회 등 추가 가능).

**YouTube Data API v3 자동 동기화 (이중입력 제거):**
- 채널(`@영천중앙교회-f6h`)이 **재생목록을 예배종류별로 운영** 중 → 재생목록 → `worshipType` 자동매핑 가능(최상 케이스).
- 동기화 흐름: `playlistItems.list`로 각 재생목록 영상 수집(videoId·제목·설명·게시일·썸네일) → `sermons`에 upsert(videoId 유니크). 호출당 1유닛/50개, 일 10,000유닛 → 비용 사실상 0.

**완전 자동 공개 (관리자 검토 없음):**
- 동작 원리: **WebSub 푸시 = 트리거**, **실제 데이터는 sync가 재생목록에서 확정**.

```
YouTube 업로드 → WebSub 푸시(콜백 라우트) → sync 실행
  → Data API로 예배 재생목록 읽기 → 신규 영상 upsert → 자동 공개(isPublished=true)
```

- 푸시 payload는 "새 영상 떴다" 신호로만 사용. worshipType 등은 sync가 재생목록 기반으로 확정.
- **WebSub(PubSubHubbub)**: 허브(`pubsubhubbub.appspot.com`)에 토픽 `youtube.com/xml/feeds/videos.xml?channel_id=...` 구독 → 새 영상마다 콜백으로 푸시. 콜백은 Next.js route handler.
- **보강 2가지 (필수)**:
  1. **구독 갱신 Cron**: WebSub 임대(lease)는 ~5일 만료. 일일 Vercel Cron이 만료 전 자동 재구독(서버→허브 POST, 사람 개입/유튜브 로그인 0).
  2. **안전망 Cron**: 푸시는 업로드 순간 발화하나 재생목록 추가가 늦으면 그 시점 worshipType 미정 + 재생목록 추가는 푸시 재발화 안 함 → 일일 Cron이 "재생목록 들어왔는데 누락된 영상" 재수거. (갱신 Cron과 통합 가능.)
- **보안**: 콜백은 구독 시 발급한 secret으로 **HMAC 서명 검증**(위조 푸시 차단).
- **사후 제어**: 관리자는 개별 영상 **숨김(unpublish)** 가능(실수 업로드 대비). 사전 검토는 없음.

**재생목록 → worshipType 매핑 (현재 채널 기준):**

| 재생목록 | 매핑 |
|---|---|
| 주일예배 | `주일예배` (설교) |
| 수요예배 | `수요예배` (설교) |
| 주일찬양예배 | `주일찬양예배` (설교) |
| 금요기도회 | `금요기도회` (설교) |
| 시온찬양대 / 특송 | 찬양(설교 아님 — 별도 카테고리 또는 제외) |
| 특별행사 | 행사 |

- 매핑 테이블은 **DB/설정으로 관리**(채널이 재생목록 추가/변경 시 코드 수정 없이 대응). 미매핑 재생목록은 기본 미공개.
- **제목 파싱(선택 강화)**: 제목에 `[주제]` 태그·날짜·설교자가 일정 형식이면 설교제목/시리즈/본문/날짜 추가 추출. 형식 불규칙 시 관리자가 검토 때 보충.
- 썸네일은 YouTube 자동 썸네일 사용(필요 시 R2 업로드).
- DB enum 정규화: `주일예배 / 주일찬양예배 / 수요예배 / 금요기도회 …`. check 제약 없이 text + 앱레벨 상수로 유연하게.

**구현 메모**: 핸들이 아니라 `channelId` 필요 → 1회 조회로 확보(또는 RSS `feeds/videos.xml?channel_id=`). `YOUTUBE_API_KEY` env 추가. 비공개/멤버십 영상은 API에 안 나옴(정상).

### 5.2 주보 (핵심 기능)

**입력 파이프라인 (하이브리드):**
1. 관리자가 hwp(또는 hwpx) 파일 업로드
2. 서버가 hwp 파싱 → 구조화 필드로 **자동 prefill**
3. 관리자가 폼에서 검토/수정 (파싱 실패 시 수동 입력 fallback)
4. 디자인된 HTML 템플릿으로 렌더 + 저장 (원본 hwp는 R2 보관)

**hwp 파싱 (검증 완료):**
- HWP 5.0 = OLE compound file. `BodyText/SectionN` 스트림은 raw deflate(zlib `-15`) 압축.
- 레코드 헤더 4바이트: `tag = hdr & 0x3ff`, `size = (hdr>>20)&0xfff` (0xfff면 다음 4바이트가 실제 size).
- **tag 67 (HWPTAG_PARA_TEXT)** 본문이 UTF-16LE 텍스트.
- 검증: `2026년 5월 31일`, `2026년 6월 7일` 주보 두 개 모두 정상 추출됨.
- 합법성: HWP 5.0 포맷 명세는 한컴이 공개(정부 공개문서 정책). 사용 라이브러리는 olefile(MIT)·zlib(표준)뿐, 한컴 SW 비의존. 교회 소유 파일을 교회 서버에서 처리 → 문제 없음.
- 파싱 라이브러리: Python(olefile) 프로토타입 검증됨. 구현 시 Node 환경에서 동등 파서 필요(순수 JS로 OLE/CFB 읽기 + inflate, 또는 별도 파싱 서비스). 구현 플랜에서 상세화.

**주보 고정 섹션** (매주 동일 구조):
- 표지: 권/호, 창립일, 발행일, 표어, 본문 말씀
- 교회정보: 주소/전화/블로그
- 주일예배 순서표 (항목/담당)
- 교회학교·장년부 예배안내 표
- 주일찬양예배(구역모임), 수요예배 순서
- 새벽기도회 본문/찬송 (요일별 표)
- 월별 "섬기는 청지기" (주차별: 주일기도/찬양/수요/안내/헌금/차량/청소)
- 헌금 명단 (십일조/감사/선교/건축/전도) + 헌금계좌
- 지난주 출석현황, 주방봉사 일정표
- 영천중앙소식 (알림/섬기는 분들/월 행사)

**개인정보 / SEO 정책:**
- 주보에는 실명 데이터 다수(헌금자·봉사자 명단, 차량/안내 담당, 출석 인원, 계좌번호).
- 본 사이트는 공개·외부 유입용 → 실명 텍스트가 검색엔진에 색인되면 PIPA 리스크 + 정서적 거부감.
- **결정: 주보 상세 페이지 전체에 `noindex`** 적용(robots meta). 정상 크롤러(구글·네이버)는 색인 제외.
  - 한계 명시: noindex는 검색 색인만 막고 스크래핑(평문 수집)은 못 막음. 단, 동일 주보가 이미 블로그에 공개 중이라 교회 수용 노출 수준과 일관됨.
  - 명단 이미지화는 **하지 않음**(모바일 반응형·접근성 유지). 추후 스크래퍼 우려 시 명단 블록만 이미지/SVG 렌더로 강화 가능.
- 유입용 SEO는 주보가 아니라 **홈/설교/소개/소식** 페이지가 담당(정상 색인).

### 5.3 갤러리
- **앨범 단위** 구조: 갤러리 = 앨범 N개, 앨범 = 사진 N장 (예: "2026 부활절", "여름 수련회").
- 이미지 업로드는 **관리자 페이지에서만**. R2 저장.
- 교회소식 하위 메뉴(top-level 아님).

### 5.4 소식·공지
- 게시판형. 카테고리(공지/소식/행사 등). 고정(pin) 가능.
- 기존 `posts` 테이블 재활용(아래 6장 참고).

### 5.5 관리자
- 로그인 후 대시보드(통계: 설교/게시글/조회수/최근 로그) + 각 콘텐츠 CRUD.
- 모든 업로드(이미지/hwp/첨부)는 관리자 페이지 경유.
- 활동 로그(`app_logs`) 기록.

### 5.6 AI 설교 요약 (선택 기능, Phase 2 이후)

설교 영상 임베드 아래 **AI 자동 요약**(3~4줄 + 핵심구절) 표시.

- 흐름: 설교 sync 시 → YouTube 한국어 자막(transcript) 추출 → Claude API로 요약 → `sermons.summary` 저장 → 페이지에 표시.
- **영상당 1회만** 생성(sync 단계, 비동기 — 요약 생성이 공개를 막지 않게). 페이지 로드마다 API 호출 안 함 → 비용 거의 0.
- 모델: 요약은 가벼워 **claude-haiku-4-5** 적합(품질 더 원하면 sonnet). 정확한 모델/단가는 구현 시 claude-api로 확정.
- 자막 의존: 자동자막 없는 영상은 요약 스킵(또는 추후 STT 옵션). 한국어 자동자막이 설교체/성경인용에 약할 수 있어 **관리자 사후 수정** 가능하게.
- **우선순위 낮음**: 필수 아님. 핵심 사이트 완성(Phase 1·2) 후 추가.

## 6. 데이터 모델 (drizzle)

기존(`src/lib/db/schema.ts`): `profiles`, `sermonSeries`, `sermons`, `posts`, `appLogs`.

**변경/추가 계획:**
- `sermons`: `worshipType` 값 정규화(주일예배/주일찬양예배/수요예배 …). videoUrl=YouTube URL.
- `posts`: 공지/소식/행사용. 주보는 별도 테이블로 분리(아래) → `category` check에서 '주보' 제거 검토.
- **신규 `bulletins`(주보)**: `id, bulletinDate(date), volume, issue, theme(표어), scripture(본문), hwpSourceUrl(R2), content(jsonb, 구조화 섹션), isPublished, createdBy, createdAt, updatedAt`.
  - 반복/표 형태 섹션은 `content` JSONB에 구조화 저장(예배순서·청지기표·헌금명단 등).
- **신규 `galleryAlbums`**: `id, title, description, coverImgUrl, eventDate, isPublished, createdAt`.
- **신규 `galleryImages`**: `id, albumId(FK), imageUrl(R2), caption, sortOrder, createdAt`.
- `profiles`: Neon Auth `users_sync`와 역할 중복 검토. 단순 모델(전원 관리자)이라 별도 profiles 최소화 가능 — 구현 플랜에서 확정.
- `appLogs`: 유지.

(정확한 컬럼/마이그레이션은 구현 플랜에서 확정. Neon은 blob 저장 불가 → 모든 파일은 R2 URL만 DB에 보관.)

## 8. 진행 단계 (Phase)

**Phase 1 — 공개 프론트(디자인) 우선, seed 데이터**
- 홈/교회소개/설교목록/주보상세/갤러리/소식 UI를 가짜·seed 콘텐츠로 완성(6/7 주보 파싱 데이터, YouTube 썸네일 자리 등 현실감 있게).
- 백엔드(DB/Auth/업로드) 미연결.
- **Vercel 프리뷰 배포** → 목사님(최종 결정자) 확인.
- 1안(따뜻한 미니멀)으로 진행, 목사님이 다른 색감/구성 원하면 변경(CSS 레벨이라 저렴).

**Phase 2 — 백엔드 연결** (디자인 승인 후)
- Neon 스키마 확정/마이그레이션, Neon Auth(가입 비활성), R2 업로드, 관리자 CRUD, hwp 파서(Node).

**Phase 3 — 마이그레이션** (본 스펙 범위 밖, 추후)
- 네이버 블로그(`ycch6645`) 과거 콘텐츠 이전. 네이버는 API/스크래핑 제약 큼 → 수동 또는 블로그 백업 export 기반 반자동. 사이트 완성 후 별도 단계.
